# AGENTS.md

Canonical agent instructions for Liatrio's Engineering Bootcamp — a Docsify-based static documentation site teaching engineering fundamentals with a DevOps focus. Read this file first for anything not obvious from the code itself.

## Domain Glossary

| Term | What it means in this repo |
| --- | --- |
| Chapter | Top-level `docs/N-*/` directory (1–11). One per major topic (fundamentals, virtualization, CI/CD, etc.). |
| Section | Sub-page inside a chapter (e.g., `docs/1-.../1.3-git.md`). Numbering appears in the filename. |
| Exercise | Hands-on unit inside a section, described in that section's YAML front-matter. |
| Front-matter | YAML block at top of a section file listing `category`, `estReadingMinutes`, `exercises[]`, `technologies[]`. Drives learning-metric visualizations. |
| Master record | Consolidated front-matter data in `docs/README.md`. Auto-generated by the pre-commit hook — never hand-edit. |
| Sidebar | `docs/_sidebar.md`. Authoritative navigation. Adding a section that is not linked from the sidebar breaks the front-matter hook. |
| Quiz | Interactive quizdown component. Source lives in `src/quizzes/chapter-N/N.X/`, embedded via a `<div class="quizdown">` wrapper. |

## Repository Layout

- `docs/` — All learner-facing content, organized by chapter. See `docs/_sidebar.md` for the authoritative section list.
- `docs/README.md` — Auto-generated master record. Regenerate with `npm run refresh-front-matter`; do not hand-edit the front-matter block.
- `docs/_sidebar.md` — Navigation. Any new section must be linked here or the pre-commit hook errors.
- `src/` — Docsify plugins, quiz sources, webpack entry points.
- `examples/` — Runnable code referenced from exercises (Go modules, docker-compose files, etc.).
- `img/` — Shared image assets. New images go here, not next to the markdown file.
- `.husky/front-matter-condenser.js` — Pre-commit hook that reconciles section front-matter into `docs/README.md`.
- `STYLE.md` — Contributor style guide (front-matter template, HTML-in-markdown rules, multi-column grids).

## Commands

Copy these verbatim; do not paraphrase.

- `npm install` — install dependencies (also runs Husky install).
- `npm start` — build dev bundle + serve Docsify on `http://localhost:3000`.
- `npm run serve:docsify` — serve only, no rebuild.
- `npm run build:dev` / `npm run build:prod` — webpack builds.
- `npm run lint` — markdownlint-cli2 over all `.md` (see `package.json` for excluded directories).
- `npm run refresh-front-matter` — regenerate `docs/README.md` from current section front-matter.
- `npm run compare-front-matter` — read-only comparison; used by pre-commit hook.

Docker:

- `docker build . -t engineering-bootcamp`
- `docker run -d -p 3000:3000 --name engineering-bootcamp engineering-bootcamp`
- `docker compose up -d` / `docker compose down` (v2 syntax — see decision table below).

There is no unit-test suite for the docs themselves. Go examples under `examples/ch8/` do have `go test`; run it when touching those modules.

## Front-matter System

Every section that is linked from `docs/_sidebar.md` must carry a YAML front-matter block matching the template in `STYLE.md`. The pre-commit hook (`.husky/front-matter-condenser.js`) walks the sidebar, reads each section's front-matter, and rewrites `docs/README.md`. If the hook aborts a commit, run `npm run refresh-front-matter`, inspect the resulting diff, and commit it together with your changes.

When adding a new exercise, reuse existing values for `category` and `technologies` from `docs/README.md` — the visualizations aggregate by exact string match, so novel casing or near-synonyms fragment the data.

## Commit & Branch Conventions

- Conventional Commits, lowercase type. Examples from history: `docs: …`, `chore: …`, `chore(deps): …`, `fix: …`, `feat: …`.
- Reference GitHub issues in the subject where applicable: `docs: update docker-compose v1 syntax to docker compose v2 (#468)`.
- Branches follow `<type>/<slug>`: `docs/move-version-control-to-ch1`, `chore/agents-md-adoption`, `fix/broken-link-chapter-4`.
- One PR per parent task / issue. Rebase on `master` before opening; never merge `master` into a feature branch.

## Content Guidelines

- Default heading level inside a page is `###` (H3). `##` (H2) appears in the Docsify TOC.
- Images use HTML `<img>` (Docsify + markdown compatibility) and live in the root `img/` folder — not co-located with the page.
- Blank line required between adjacent HTML and Markdown blocks (Docsify parser requirement — see `STYLE.md`).
- Multi-column layouts use `<div class="grid2|grid3|grid4">` with `<div class="col">` children.
- Quiz embed: place source at `src/quizzes/chapter-N/N.X/<slug>.js`, then embed with:

  ```html
  <div class="quizdown">
    <div id="chapter-N/N.X/<slug>.js"></div>
  </div>
  ```

  The `id` is the path relative to `src/quizzes/`. Do not inline `<script>` tags or `import` statements.

## Decision Tables

When multiple valid patterns exist, pick per this table.

| Situation | Use | Do not use | Why |
| --- | --- | --- | --- |
| New Docker Compose invocation in prose or code block | `docker compose <verb>` (v2, space) | `docker-compose <verb>` (v1, hyphen) | v1 unsupported since 2023. Filenames (`docker-compose.yaml`) stay hyphenated. |
| Adding a category or technology to front-matter | Reuse an existing string from `docs/README.md` | Introduce a new synonym / different casing | Visualizations aggregate by exact match; fragmentation kills the charts. |
| Referencing an image | Root `/img/foo.png` via `<img>` | Chapter-local path or Markdown `![]()` | Keeps assets in one place and preserves HTML/markdown interop. |
| Cross-linking a section | Relative path from current file (`../3-.../3.2-foo.md`) | Absolute Docsify hash (`#/3-.../3.2-foo`) | Relative paths survive `git mv`; hash paths silently rot. |
| Documenting a new tool or command in an exercise | Add to that section's `technologies:` list | Freeform prose only | Feeds the tech word-cloud. |
| Referencing another repo doc from an agent | Plain path with "read when needed" | `@path` auto-import | Auto-imports expand at every session start; only this file uses the stub pattern. |

## Do / Don't

- Do run `npm run lint` before pushing. Don't rely on CI to catch markdownlint failures.
- Do add every new section to `docs/_sidebar.md` in the same commit that creates the file. Don't ship the file first and update the sidebar later — the pre-commit hook will refuse the next commit.
- Do let the pre-commit hook regenerate `docs/README.md`. Don't hand-edit the master-record front-matter block.
- Do reuse existing category/technology strings. Don't invent variants ("Kubernetes" vs "K8s" fragments the data).
- Do preview changes with `npm start` and click through affected pages. Don't assume markdown-only edits render correctly — Docsify has parser quirks around HTML.
- Do put screenshots and diagrams under `img/` at the repo root. Don't co-locate images next to the markdown file.
- Do keep code fences small (≤10 lines) — link to files under `examples/` for anything longer.
- Do write commit subjects that name the artifact and the reason. Don't ship subjects like "update" or "fixes".

## Destructive-Action Guardrails

- Never push directly to `master`. All changes land via PR from a feature branch.
- Never rewrite history on `master` (`push --force`, `reset --hard`, filter-branch). Feature-branch force-push is fine before review.
- Never bypass the Husky pre-commit hook (`git commit --no-verify`) to work around front-matter validation. Fix the front-matter or run `npm run refresh-front-matter` and commit the regenerated master record.
- Never hand-edit `docs/README.md`'s front-matter block; it is machine-generated.
- Never delete a chapter or renumber sidebar entries without also fixing every cross-link (`grep -rnE '<old-path>' docs/`).
- Before deleting `examples/` code, confirm it is not referenced from any `docs/**/*.md` file.

## Keeping This File Current

Update this file whenever a command name, directory structure, or convention changes — an out-of-date AGENTS.md is worse than a short one. Aim for ~150 lines; the hard ceiling is 300. If a section grows past ~30 lines, consider moving detail to a dedicated doc under `docs/` and leaving a "read when …" pointer here.
