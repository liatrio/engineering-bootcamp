# 9.6 KEDA

[KEDA](https://keda.sh/) (Kubernetes Event-Driven Autoscaler) is an open-source project that extends Kubernetes with the ability to scale workloads based on signals from external event sources — message queues, databases, HTTP request rates, and more. Where a native HPA reacts to resource metrics like CPU and memory, KEDA can react to the thing that actually drives resource consumption: the work waiting to be done.

## Exercise 1 — Meet the System

E-commerce startup, Bugs R Us has recruited you as their sole devops engineer. They have their ordering system deployed to Kubernetes. However, they haven't implemented any autoscaling and are expecting a large influx in orders next week when they have a flash sale on dried crickets. Resources have been highly constrained for the startup after the CEO lost the annual budget on a meme coin. Their Redis database can only hold a limited number of orders before it runs out of memory and orders begin to get dropped. It's up to you to architect a scaling solution that will prevent the database from getting overwhelmed during the flash sale, but can also scale back down once traffic dies down, so they don't go further into debt.

The manifests and supporting infrastructure for their cluster can be found in the `examples/ch9/keda` directory of the [bootcamp repository](https://github.com/liatrio/engineering-bootcamp).

The system has two services:

- **checkout-service** — an HTTP service that accepts orders from customers and pushes them onto a Redis queue
- **order-processor** — a worker that pops orders from the queue and processes them one at a time

![producer consumer image](img9/producer-consumer.jpg ':class=img-center :alt= producer consumer image')

This producer/consumer pattern is one of the most common architectures in distributed systems. It decouples the rate at which work arrives from the rate at which it is processed, allowing each service to be scaled and deployed independently.

Take some time to explore the project and understand its structure. You don't need to understand every line, but knowing how the different components of the system interact will be critical to completing this exercise. In particular, look at the [Taskfile](https://taskfile.dev/docs/guide) and get a feel for how it can help orchestrate the cluster.

Once you have a grasp on how the system works, stand up a local instance of it.

## Exercise 2 — HPAs

The HPA controller evaluates scaling decisions on a fixed interval — 15 seconds by default, controlled by the `--horizontal-pod-autoscaler-sync-period` flag on the kube-controller-manager. The CPU metrics it reads come from the metrics-server, which scrapes each node's kubelet every 60 seconds by default. This means a scaling decision can be based on data that is over a minute old, and the HPA may not act on a sudden spike until well after it has occurred. For this exercise, the Taskfile has already configured the cluster with a 5-second sync period and a 15-second metrics-server resolution to make scaling behavior more observable. This is important because the load generator script that simulates the flash sale only runs for a minute.

1. Now that we have a local instance running, run the included load generator script which is supposed to simulate a flash sale. This can be done with either the Taskfile or with a raw [`k6`](https://grafana.com/docs/k6/latest/examples/get-started-with-k6/test-for-performance/) command.

2. Now observe how the effect this has. The Taskfile has a simple observability solution included.

?> In a real enterprise system observability is generally handled with an exporter and a dashboard like the Prometheus + Grafana stack.

3. What can we observe when the load generator runs? Can the single pod keep up with the sudden influx in orders?

4. Try fixing the issue with an HPA that scales based on CPU load. Does this work?

5. What is the fundamental limitation of using CPU as a scaling signal for this workload?

## Exercise 3 — KEDA

Recall that unlike vanilla HPAs KEDA can scale in response to external events. What external events would be good candidates for scaling here?

Before installing KEDA, you must remove the HPA you created in the previous exercise. Only one autoscaler can manage a given Deployment at a time; leaving both in place will cause a conflict.

1. Delete your HPA and confirm it is gone before continuing.

2. Install KEDA to your cluster.

3. Verify that the KEDA controller pods are `Running` in the `keda` namespace before continuing.

4. Now choose an external event that will trigger scaling and try using KEDA to solve the problem.

### Deliverables

- Why did the native HPA fail to protect the system under a sudden surge, even after it scaled out?
- In the producer/consumer pattern, what are the tradeoffs of scaling the consumer based on queue depth vs. scaling it based on CPU utilization?
- Read about [proactive scaling](https://netflixtechblog.com/scryer-netflixs-predictive-auto-scaling-engine-a3f8fc922270) and discuss how it differs from reactive scaling. Is one better than the other?
- Aside from queue depth, what metrics can determine how much you should scale?
- Why not set metrics servers resolution and HPA sync period to something even shorter to maximize the responsiveness of your scaling?
