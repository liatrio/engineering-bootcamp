# 9.6 KEDA

KEDA (Kubernetes Event-Driven Autoscaler) is an open-source project that extends Kubernetes with the ability to scale workloads based on signals from external event sources — message queues, databases, HTTP request rates, and more. Where a native HPA reacts to resource metrics like CPU and memory, KEDA can react to the thing that actually drives resource consumption: the work waiting to be done.

This distinction matters most for bursty, queue-driven workloads. CPU is a lagging indicator: by the time a worker pod's CPU spikes, it has already been processing a backlog for some time. Queue depth is a leading indicator — it reflects exactly how much work is waiting right now. Scaling on queue depth lets a system respond before pressure builds rather than after.

This section builds on the HPA concepts from 9.5 using a companion e-commerce example in `examples/ch9/keda/`. You will deploy a realistic producer/consumer system, observe it fail under load, attempt a fix with a native HPA, and then replace it with KEDA's event-driven autoscaling.

## Exercise 1 — Meet the System

The example system has two services:

- **checkout-service** — an HTTP service that accepts orders from customers and pushes them onto a Redis queue
- **order-processor** — a worker that pops orders from the queue and processes them one at a time

This producer/consumer pattern is one of the most common architectures in distributed systems. It decouples the rate at which work arrives from the rate at which it is processed, allowing each service to be scaled and deployed independently.

1. From `examples/ch9/keda/`, run `task deps:check` and resolve any missing tools before continuing.

2. Use the Taskfile to create the cluster, build the images, and deploy the full stack. Run `task --list` to see all available tasks.

3. Verify that the `checkout-service`, `order-processor`, and `redis` pods are all `Running` in the `keda-demo` namespace before moving on.

## Exercise 2 — Simulate Peak Traffic

1. Open a second terminal and start the observer with `task observe`. Keep this running throughout all remaining exercises — it shows queue depth, pod count, and average CPU in real time.

2. In your first terminal, run `task load:run` to simulate a flash sale.

3. Watch the queue depth climb. Does the pod count change? Note the peak queue depth — you will compare this against later runs.

## Exercise 3 — Attempt a Fix with Native HPA

The order-processor performs CPU-bound work for each order. Kubernetes' built-in HPA can scale a Deployment based on CPU utilization — let's see if that's sufficient.

1. Inspect the order-processor Deployment manifest. What CPU resources are configured? Using the Kubernetes documentation, write an HPA that targets the order-processor and scales based on CPU utilization, then apply it to the cluster.

2. Re-run `task load:run` while `task observe` is running.

3. Does the HPA scale the Deployment out? If so, how quickly? How does the peak queue depth compare to Exercise 2?

4. What is the fundamental limitation of using CPU as a scaling signal for this workload?

## Exercise 4 — Introduce KEDA

KEDA can scale the order-processor based on how many unprocessed orders are sitting in the queue — before CPU pressure ever materializes.

Before installing KEDA, you must remove the HPA you created in Exercise 3. Only one autoscaler can manage a given Deployment at a time; leaving both in place will cause a conflict.

1. Delete your HPA and confirm it is gone before continuing.

2. Install KEDA into your cluster using Helm. The KEDA project maintains an official chart at `kedacore/keda` — use version `2.16.0` for this exercise.

3. Verify that the KEDA controller pods are `Running` in the `keda` namespace before continuing.

## Exercise 5 — Create a ScaledObject

A `ScaledObject` is the KEDA resource that connects a scaling trigger to a Kubernetes workload. Before writing one, review the manifests that were deployed with your stack — pay close attention to `keda-trigger-auth.yaml` and what role it plays.

Using the [KEDA documentation](https://keda.sh/docs/), write a `ScaledObject` that:

- Targets the `order-processor` Deployment
- Uses the appropriate scaler for a Redis list queue
- Sets a minimum and maximum replica count appropriate for this workload
- Uses a queue depth threshold that reflects how many messages a single replica can realistically keep up with

Apply your `ScaledObject` and run `kubectl get scaledobject -n keda-demo`. The `READY` column should show `True` before you proceed.

## Exercise 6 — Run the Load Test Again

1. With `task observe` running, re-run `task load:run`.

2. Watch the queue depth and pod count. How does the system behave compared to Exercises 2 and 3? At what queue depth does KEDA begin scaling out?

3. Compare the peak queue depths across all three runs. What does the difference illustrate about reactive versus proactive scaling?

### Deliverables

- Why did the native HPA fail to protect the system under a sudden surge, even after it scaled out?
- What does KEDA use as its scaling signal in this exercise, and why is it a more appropriate metric than CPU for this workload?
- In the producer/consumer pattern, what are the tradeoffs of scaling the consumer based on queue depth vs. scaling it based on CPU utilization?
